## [Unreleased]

- **Added**
  - **Unit-only enum support in `#[derive(clapfig::Schema)]`** ([#54](https://github.com/arthur-debert/clapfig/issues/54)) — A field of an enum type (`page_size: PdfPageSize`) deriving `Schema` now flattens to `LeafType::Enum { values: [...] }` automatically; the `#[clapfig(value)]` workaround is no longer required. `SchemaStatic` gains `enum_variants: &'static [&'static str]`; the static→runtime converter inspects this to decide between `Field::Nested` (object) and `Field::Leaf(LeafType::Enum)`, so the macro never has to syntactically distinguish struct vs enum at the field position. Payload variants are rejected at derive time pointing at `#[clapfig(value)]`. Variants honor `#[clapfig(rename_all = "...")]` / `#[serde(rename_all = "...")]` (lowercase, UPPERCASE, PascalCase, camelCase, snake_case, SCREAMING_SNAKE_CASE, kebab-case, SCREAMING-KEBAB-CASE) with per-variant `#[clapfig(rename = "...")]` / `#[serde(rename = "...")]` overriding the global rule. `config gen` emits `# Allowed:` and JSON Schema emission carries the `enum` array, same as the runtime path.
  - **`Schema::field_paths()` — auto-generated dotted-path inventory** ([#54](https://github.com/arthur-debert/clapfig/issues/54)) — New default method on the `Schema` trait that walks the macro-emitted `SchemaStatic` and returns every dotted path in the schema (leaves + sections) in source order. Lets downstream consumers replace hand-maintained "known paths" constants — new fields show up automatically. Enum-kind nested schemas surface as one leaf path (variants are leaf metadata, not sub-paths); array-of-objects nodes contribute the array name only (entries aren't addressable at this layer). Free helper `static_schema::collect_field_paths(&SchemaStatic)` exposed for callers that hold the static directly. No macro change required — the default impl reads `Self::STATIC`.
  - **`UnknownKeyDecision::Collect` + `load_with_unknowns()`** ([#54](https://github.com/arthur-debert/clapfig/issues/54)) — Closes the gap that forced LSP-style hosts wanting both typo-rejection *and* extension-key visibility to smuggle a shared `Vec` through an `on_unknown_key` closure capture. The new `Collect` variant, when returned for a cascade-strict key, appends to a collected-unknowns list returned alongside the typed config rather than failing the load. `Reject` and `Accept` keep their existing semantics. New `CollectedUnknown` struct (owned: path, leaf, value as `toml::Value`, file, line). New `load_with_unknowns()` on `ClapfigBuilder`, `RuntimeBuilder`, and `SchemaConfigBuilder` returns `(C, Vec<CollectedUnknown>)`; existing `load()` is unchanged. New `resolve_at_with_unknowns()` on `Resolver<C>` and `RuntimeResolver` for tree-walk callers.
  - **`Builder::accept_dotted_extension_keys_in(path, decision)`** ([#54](https://github.com/arthur-debert/clapfig/issues/54)) — Built-in for the "accept dotted, reject bare" unknown-key pattern (the closure that the issue's author and lex-fmt PR #658 both wrote by hand). Under `path`, an unknown key whose raw TOML leaf contains a `.` (a `[serde(flatten)]`-style schema-extension key like `"acme.task-due-date-missing"`) is resolved with `decision` — typically `Accept` for CLI hosts that drop them silently, or `Collect` for LSP hosts that surface them via `load_with_unknowns()`. Bare-leaf unknowns still fall through to `Reject`. `path` is bounded by segment (`"diag"` won't match `"diagnostics.rules"`); pass `""` to apply everywhere. Composes with `strict_at`: lenient-cascade keys drop before the helper sees them. Available on `ClapfigBuilder`, `RuntimeBuilder`, and `SchemaConfigBuilder`; the closure-as-primitive shape is unchanged for callers with nuanced needs. Known limitation: on the static (confique) path, `serde_ignored` collapses TOML quoted-key dots before the callback sees them, so the helper's "leaf contains a dot" predicate fires only on the runtime / `#[derive(Schema)]` paths in practice.
  - **`Field::MapOf(Schema)` — maps of nested objects on the runtime path** ([#54](https://github.com/arthur-debert/clapfig/issues/54)) — Dual of `ArrayOf` for keyed maps. Builder method `Schema::object(...).map_of(name, item_schema)`. `SchemaRef`/`FieldKindRef::MapOf` exposes it to the schema walkers (cascade strictness, JSON Schema, doc lookup, field-paths inventory). Map entries recurse for unknown-key collection, default filling, required-field checks, and per-leaf type validation; entry paths are `parent.entry_key.field` and the cascade walks them like `ArrayOf`'s bracketed form. JSON Schema emits `type: object` with `additionalProperties: <item schema>`. Derive macro: `{Hash,BTree}Map<String, NestedStruct>` now emits `FieldStatic::MapOf(<T as Schema>::STATIC)` instead of erroring — drops one of the four `#[clapfig(value)]` workarounds from the lex-fmt feedback.

- **Fixed**
  - **Env-derived unknown keys route through the strict-mode cascade** ([#54](https://github.com/arthur-debert/clapfig/issues/54)) — Before this change, `validate_unknown` only ran on the per-file pass; the env layer landed on the merged table afterwards and `finalize` doesn't re-check unknown keys. The observable effect was that an env var like `MYAPP__ROGUE_KEY=1` reached `extra` (via `#[serde(flatten)]`) or silently sat on the merged table without ever hitting the cascade or the `on_unknown_key` callback. New `validate::collect_unknown_paths_ref` is a schema-driven (not serde-driven) unknown-key walker that works over any `SchemaRef` (static or runtime), skips `ArrayOf` subtrees (no env / CLI / URL dotted syntax can build them), and recurses through `Nested` and `MapOf`. `resolve::resolve` now runs it on the env-derived table when the cascade is active, threading the result through `filter_through_cascade`. Synthesized source path is `<env>` (rendered in error output); source text is empty (no line number). The walker is intentionally schema-only — running `serde_ignored` here would surface env's heuristic value parsing (string vs int vs float) as type errors that `finalize`'s typed-deserialize step already covers. Pre-Phase-3 `strict(false)` remains byte-identical (the new pass is gated on `cascade_active`, same as the file-level walk).
  - **`#[clapfig(rename_all = ...)]` acronym handling matches serde** — The previous `pascal_to_snake` / `pascal_to_camel` rewrites emitted `MyHTTPApi → my_h_t_t_p_api` (every letter separated). Serde's actual output is `my_http_api` (acronym runs collapse). Two boundaries now insert the separator: (a) lowercase→uppercase, (b) uppercase→uppercase when the second upper is followed by a lowercase letter (i.e. the second upper starts a new word inside an acronym-then-PascalCase sequence). `camelCase` derives from the snake form so it stays consistent. No `heck` dependency — single text transform.
  - **`Option<NestedStruct>`, `Option<EnumType>`, `Option<Map<String, NestedStruct>>` produce diagnostics instead of macro panics** — Before this commit those classified as `Optional(Nested(_))` / `Optional(MapOfNested(_))` and hit an `unreachable!` arm inside `inner_leaf_type`, panicking the proc macro instead of producing a user-facing error. New early-exit in `expand_field` returns a `syn::Error` with guidance.

- **Changed**
  - **`validate.rs` `fields` is now `Vec<(&str, FieldKindRef)>`** — Drops the per-field `String` allocation in the inner walker; comment now matches the implementation (linear scan; `HashMap` remains a follow-up if a hot path appears).
  - **`resolve_at` / `resolve_at_with_unknowns` deduped through a private `resolve_at_inner`** — Same dispatch shape on both the static (`Resolver<C>`) and runtime (`RuntimeResolver`) paths; the no-unknowns surface drops the second tuple element.
  - **Derive "struct attribute" diagnostics renamed to "type attribute"** — The same parser now handles unit enums; the trybuild stderr fixture for `unknown_struct_attr` is updated.

## [0.21.2] - 2026-05-20


- **Fixed**
  - **`#[clapfig(value)]` works on `#[serde(untagged)]` enums and other non-scalar field types** — The macro previously rejected the `value` escape-hatch attribute on any field whose Rust type fell through to `TypeShape::Nested` (the classifier's fallback for custom Pascal-case types). That branch's "leaf attributes are not valid on nested-struct fields" rejection fired before `force_value` was considered, blocking the attribute's primary documented use cases: `#[serde(untagged)]` enums, plain custom Rust enums, primitives clapfig doesn't recognize as scalars (`char`), and `BTreeMap<String, CustomStruct>`. The fix hoists the `force_value` check above the `TypeShape::Nested` rejection and skips `classify_type` entirely when the attribute is set — the user is explicitly opting out of shape inference, so we must not constrain which Rust types can carry the override. Outer `Option<T>` is still detected (syntactically only, no inner recursion) so `Option<CustomEnum>` with `#[clapfig(value)]` produces an optional `Value` leaf. New behavioral coverage in `tests/derive_schema_depth.rs` (eight tests covering `char`, custom enums, untagged enums, `BTreeMap<String, CustomStruct>`, and `Option<CustomEnum>`).

- **Added**
  - **`#[derive(clapfig::Schema)]` — typed static path with full schema-metadata symmetry** ([proposal](docs/proposals/schema-metadata-symmetry.md)) — New procedural macro (sibling `clapfig-derive` crate, gated behind the opt-in `derive` Cargo feature) emits a const-friendly `SchemaStatic` tree and implements the new `clapfig::Schema` trait on the user's config struct. New `Clapfig::schema_builder::<C>()` entry point returns a `SchemaConfigBuilder<C>` with the same surface as `ClapfigBuilder<C>` (the confique-derived static path) and `RuntimeBuilder` (the dynamic-schema path) — `app_name`, `search_paths`, `env_prefix`, `cli_override`, `post_validate`, `load`, `handle`, etc. — and produces a typed `C`. Internally wraps `RuntimeBuilder` with the cached schema shared via `Arc<Schema>` (no per-builder allocation), so every existing schema consumer — JSON Schema, `config gen`, persistence validation, strictness cascade, `LeafType` value validation — sees the same metadata the runtime path does. Closes the four observable gaps from the proposal: JSON Schema `"type"` is emitted for every leaf (including those without defaults); JSON Schema `"enum"` is emitted for `#[clapfig(allowed = [...])]` leaves; `config gen` emits `# Allowed: ...` lines for enum leaves and `#key = <placeholder>` for required leaves without a default; `config set` / load value validation runs through `LeafType::check` for crisp `ClapfigError::InvalidValue` messages.

    **Supported field types:** every Rust integer (`i8`/`i16`/`i32`/`i64`/`u8`/`u16`/`u32`/`u64`/`usize`/`isize` — all map to TOML's signed-64-bit integer with the `i64::MAX` caveat on the unsigned variants), `f32`, `f64`, `bool`, `String`, `toml::value::Datetime`, `toml::Value`, `Option<T>`, `Vec<T>` (where T is a scalar), `HashMap<String, V>` and `BTreeMap<String, V>` (where V is a scalar / value / array leaf), nested structs that also derive `clapfig::Schema`. Rejected at derive time with targeted diagnostics: `i128`/`u128` (TOML width); generic structs (`static SchemaStatic` can't reference type parameters); `Option<Option<T>>`; non-`String` map keys; `#[clapfig(allowed = [])]`; type-mismatched `allowed` entries; `value` + `allowed` on the same field; leaf attrs on nested-struct fields. Trybuild compile-fail cases (`tests/ui/derive/*.rs`) lock the diagnostics.

    **Field attributes:** `#[clapfig(default = ...)]` (accepts string / integer / float / bool literals, unary-negated numeric literals including `i64::MIN`, array literals on `Vec<T>` fields, and datetime-string literals on `toml::value::Datetime` fields — datetime literals are *not* parsed at derive time so a malformed literal panics on first `Schema::schema()` call), `#[clapfig(env = "NAME")]`, `#[clapfig(rename = "name")]`, `#[clapfig(value)]` (escape hatch — applies to any field type, the caller takes responsibility for the deserialize side; matches the `#[serde(untagged)]` enum use case), `#[clapfig(allowed = [...])]` (works on any scalar leaf, literal types must match the field's TOML kind, accepts negative integer/float literals), `#[clapfig(optional)]`. Struct-level: `#[clapfig(name = "Name")]`, `#[clapfig(strict = true/false)]`.

    **Migration from `#[derive(confique::Config)]`:** the confique path is **not** deprecated yet — existing users keep working byte-identically. To opt into the new macro for a config struct:

    1. In your `Cargo.toml`, add the `derive` feature: `clapfig = { version = "...", features = ["derive"] }`.
    2. On the struct definition, change `#[derive(confique::Config)]` to `#[derive(clapfig::Schema, serde::Serialize, serde::Deserialize)]`. The new derive does *not* generate a serde-side `Deserialize` impl — you derive that yourself (most existing confique users already do for round-tripping).
    3. Rename field attributes: `#[config(default = ...)]` → `#[clapfig(default = ...)]`, `#[config(env = "...")]` → `#[clapfig(env = "...")]`. Drop `#[config(nested)]` (the new macro infers nesting from the field type). If you used `#[config(deserialize_with = ...)]`, move it to `#[serde(deserialize_with = ...)]` on the same field.
    4. Change your entry point from `Clapfig::builder::<C>()` to `Clapfig::schema_builder::<C>()`. The builder surface is the same; `load()` still returns `C`; `post_validate(|c: &C| ...)` is still typed.
    5. Built-in confique features without a clapfig equivalent are explicit gaps for now: `#[config(partial_attr(...))]` (Layer-struct customization), `#[config(validate = ...)]` (field-level validators — use `post_validate` instead). If you depend on them, stay on `Clapfig::builder::<C>()` for that struct.

    A future PR will accept `#[config(...)]` field attributes inside the new macro as a transition convenience, reducing the migration to "rename the derive." Tracked in the proposal's open questions.
  - **`LeafType::Value` — runtime-schema escape hatch for union-shape leaves** ([#47](https://github.com/arthur-debert/clapfig/issues/47)) — New `Field::value()` constructor on the runtime path that produces a `LeafType::Value` leaf accepting any TOML value (scalar, array, table). Closes the asymmetry where the static path can express "either a bare string *or* an array of `[string, table]`" via serde's `#[serde(untagged)]`, but the runtime path had no equivalent: `LeafType::Array(String)` would reject the bare form, `LeafType::Map(String)` would reject both, and `LeafType::Enum` is scalar-only. With `Field::value()`, runtime-schema callers (e.g. lex-fmt's `[diagnostics.rules]` where one rule may be `missing_footnote = "warn"` and another `table_inconsistent_columns = ["warn", { max_columns = 80 }]`) can declare the union shape inline; clapfig accepts any shape at the schema layer, and the caller does further validation via `serde::Deserialize` inside a `post_validate` hook or after `RuntimeResolver::resolve`. `config gen` emits a `# Accepts: any TOML value` hint above the leaf line; JSON Schema follows convention and omits the `type` key entirely (no constraint). Strict mode is unaffected — `Value` is about *value shape* on a known key, not about whether unknown sibling keys are allowed. Strictly additive: no existing schema breaks because no existing schema uses the new variant.

- **Changed**
  - **Strict cascade activates validation whenever any override is `strict = true`** — Pre-Phase-3 `.strict(false)` skipped strict-mode validation entirely, with the side effect of masking type errors in config files (they would surface later in confique's typed-deserialize step). Combining a lenient default with at least one strict override (`.strict(false).strict_at("X", true)`) now activates the validation step, so type errors anywhere in the file can surface as `ParseError`. Plain `.strict(false)` with no `strict_at(_, true)` and no per-node `Schema::strict(true)` remains byte-identical to the pre-Phase-3 behavior. See `docs/strictness.md` for the decision chain.
- **Added**
  - **Cascading strictness** ([#37](https://github.com/arthur-debert/clapfig/issues/37)) — Three composable knobs decide whether an unknown key is a strict-mode violation. Defaults preserve today's behavior; everything is additive. (1) The existing `ClapfigBuilder::strict(bool)` keeps its meaning — the whole-resolution default. (2) New `ClapfigBuilder::strict_at(path, bool)` and `RuntimeBuilder::strict_at(path, bool)` set a per-section override at a dotted path; the cascade rule then resolves any unknown key by walking up to the nearest explicit ancestor (or the builder default if none). Targeting a leaf or an unknown path errors at `build_resolver` time as the new `ClapfigError::InvalidStrictPath { path, reason }`. `strict_at` honors `normalize_keys(true)` — the path can be written in kebab-case. (3) New `ClapfigBuilder::on_unknown_key(fn)` and `RuntimeBuilder::on_unknown_key(fn)` register a per-key callback that runs *only* on cascade-rejected keys; returning `UnknownKeyDecision::Accept` drops the key silently, `UnknownKeyDecision::Reject` produces a `ClapfigError::UnknownKeys` entry. The callback receives an `UnknownKeyContext` with the dotted path, the raw TOML leaf key (preserves quoted-key semantics so a literal `"acme.task-due-date-missing"` stays intact), the parsed value, the source file, and the 1-indexed line number. Composes with the cascade for sibling-level catch-alls (lex-fmt #39 style: typed fields and a `#[serde(flatten)] BTreeMap` share a struct level — the callback can match on `leaf.contains('.')` to accept extension-emitted dotted keys while rejecting bare typos). Runtime schemas additionally set per-node strictness via `Schema::strict(bool)` (the field exists from Phase 2; the cascade consumes it here); builder `strict_at` overlays runtime `Schema::strict` when both target the same path. New public exports: `UnknownKeyContext`, `UnknownKeyDecision`.
  - **Runtime-defined schemas** ([#36](https://github.com/arthur-debert/clapfig/issues/36)) — New `clapfig::runtime` module with owned `Schema`, `NamedField`, `Field`, `Leaf`, and `LeafType` types plus a fluent builder, for callers without a compile-time `confique::Config` struct. `LeafType` covers TOML primitives + array + map + `Enum { values }` for constrained value sets (log levels, output formats, modes). New `Clapfig::runtime(schema)` entry point returns a `RuntimeBuilder` with the same surface as `ClapfigBuilder<C>` — `app_name`, `search_paths`, `env_prefix`, `cli_override`, `cli_overrides_from`, `post_validate`, `build_resolver`, `load`, `handle`, etc. — but produces a `toml::Table` instead of a typed struct. `post_validate` receives `&toml::Table`. New `RuntimeResolver` parallels `Resolver<C>` for tree-walk use cases. Every doc-consuming surface (`config gen`, `config get`, JSON Schema `description`/`enum`, `meta::doc_for_runtime`) works on the runtime path with no extra wiring: `config gen` emits a fully commented template including allowed values for enum leaves (e.g. `# Allowed: "debug" | "info" | "warn" | "error"`); the JSON Schema emitter renders the same set as `enum: [...]` on the property. New `ClapfigError::MissingRequired { key }` variant produced when a required runtime leaf has no merged value and no default. New public exports: `runtime::*`, `RuntimeBuilder`, `RuntimeResolver`. Example: `examples/runtime_schema/main.rs`.
  - **`ConfigSpec` + `SchemaRef` internal abstraction** ([#35](https://github.com/arthur-debert/clapfig/issues/35)) — Internal refactor with no user-visible API change. Introduces `SchemaRef<'a>` (borrowed read-only view) and a `ConfigSpec` trait so consumers that today key off `C::META` (`validate::validate_unknown_keys`, `meta::doc_for`, `overrides::valid_keys`, `ops::generate_template`, `schema::generate_schema`, persistence) consume the abstraction instead. Static path ships `StaticSpec<C>`; runtime path ships `DynamicSpec` (see above). `ResolveInput` and `resolve` become generic over the spec. All types stay crate-private until the runtime path needs them.
  - **`clapfig::meta::doc_for::<C>(key)` for programmatic doc-comment lookup** — New thin accessor that returns the `///` doc-comment lines for a config struct's field by dotted key path, walking confique's static `META` tree directly. Returns `Option<Vec<String>>` so callers can distinguish "key doesn't exist" from "key exists, undocumented." Key spelling is lenient: `database.pool-size` and `database.pool_size` find the same field regardless of whether the builder has `.normalize_keys(true)` set. Previously, the only ways to get this data were `ConfigAction::Get` (which runs the full resolve pipeline just to attach the doc) or generating the JSON Schema and walking it — both heavyweight for tools that just want `--describe`, tooltips, or generated help text. Internal callers (`get_value`, `get_scope_value`) now delegate to the same helper, dropping the duplicate private `lookup_doc` walker.
- **Fixed**
  - **Generated TOML template now respects `.normalize_keys(true)`** — Previously, a builder configured with `.normalize_keys(true)` would still emit snake_case keys in the template produced by `config gen` (and `ConfigAction::Gen`), telling users they could type `pool-size = 5` while showing them `pool_size = 5` in the very template they were meant to copy from. The template generator now rewrites section headers and key lines to kebab-case when the flag is on, while leaving doc comments and the value portion of every line untouched. `config set` and the file-seed path that backs it still emit snake_case keys for now — wiring kebab through the persist/set path is a sibling fix.
- **Added**
  - **Kebab-case key normalization** — New `ClapfigBuilder::normalize_keys(bool)` builder method (opt-in, default `false`). When enabled, every key crossing the boundary into clapfig — TOML table keys in config files, dotted keys in `.cli_override()` / `.cli_overrides_from()`, URL query parameter keys — has its `-` characters rewritten to `_` before validation, merging, and deserialization. The user can then write `pool-size = 5` in a config file (or `--set database.pool-size=5` on the CLI) and have it map to a Rust field named `pool_size`. snake_case keys continue to work unchanged; this is purely additive. Strict-mode validation still flags genuine unknown keys, just in their normalized form, and the line-number heuristic learns to match keys regardless of dash/underscore spelling so error snippets still point at the user's original line. Environment variables are unaffected — shells dislike `-` in variable names, and the env layer already lower-cases segments and uses `__` as the nesting separator. The pipeline is now strictly parse → normalize → validate → merge, so key strings supplied by the user are never used directly without the normalization step first. Two distinct keys in the same table that collapse to the same name (e.g. `pool-size` and `pool_size`) surface as a new `ClapfigError::NormalizedKeyCollision` rather than silently dropping one entry.
## [0.17.0] - 2026-04-28

- **Documentation**
  - **Crate-level guide covers `post_validate` and `Resolver<C>`** — Added two new sections to the `lib.rs` crate doc (which is the front page of docs.rs/clapfig): "Semantic validation — the `post_validate` hook" explains why structural validation isn't enough and documents the hook's signature, short-circuit semantics, composition rules, and design tradeoffs; "Tree-walk resolution — the `Resolver<C>` handle" documents the `.htaccess`/`.editorconfig` use case, the per-call `Cwd`/`Ancestors` anchoring model, the instance-scoped file cache's no-mtime-check contract, and the `load() = build_resolver()?.resolve_at(cwd)?` relationship. `Resolver`'s type-level rustdoc links out to the crate-level section rather than duplicating the design doc, and all intra-doc links now use fully-qualified paths so docs build warning-free. README features list gains bullets for tree-walk resolution, the post-merge validation hook, and JSON Schema generation.
- **Added**
  - **`Resolver<C>` and tree-walk resolution** — New `ClapfigBuilder::build_resolver()` method returns a reusable `Resolver<C>` handle that can be called repeatedly with `.resolve_at(&dir)` to produce a typed configuration anchored at a specific directory. Unlocks the `.htaccess` / `.gitignore` / `.editorconfig` pattern where every directory in a dynamic file tree is its own resolution root. `SearchPath::Cwd` and `SearchPath::Ancestors` are interpreted relative to the directory passed to each `resolve_at` call instead of the process's current working directory, so walking a content tree yields one independently-merged config per leaf. Files read during resolution are cached inside the resolver by absolute path, so tools that walk a 1000-leaf tree sharing five ancestor config files pay the disk+parse cost once per unique file instead of 1000×. Any `.post_validate()` hook registered on the builder is captured into the resolver and fires on every `resolve_at` call. New public export: `Resolver`.
  - **`.post_validate()` hook** — New builder method `ClapfigBuilder::post_validate(|c: &C| -> Result<(), String>)` registers a closure that runs after all layers have been merged and confique has type-validated the result, but before `load()` returns. Use it for constraints confique can't express: numeric ranges ("port must be ≥ 1024"), cross-field invariants ("if A is set then B must be set"), enum combinations, filesystem preconditions, or anything that depends on the merged value. Rejection messages are wrapped in the new `ClapfigError::PostValidationFailed(String)` variant. The hook runs only when upstream resolution (parsing, strict-mode validation, type-checking) succeeds; failures short-circuit before it fires. Calling `.post_validate()` more than once replaces the previous hook.
  - **JSON Schema generation** — New `clapfig::schema::generate_schema::<C>()` walks confique's `Meta` tree to produce a JSON Schema (Draft 2020-12) document describing the config struct. Each nested struct becomes an `object` with `properties`; non-`Option<T>` fields are listed in `required`. Leaf types are inferred from `#[config(default = ...)]` expressions (string, integer, number, boolean, array, object), doc comments become `description`, defaults are emitted as `default`, and env var names are attached as the `x-env` extension. Intended for auto-generating UI editors, external validation tools, and IDE integrations. Exposed via `app config schema` (with `-o/--output` to write to a file) on both `ConfigArgs` (derive) and `ConfigCommand` (runtime builder, renameable via `.schema_name()`). New `ConfigAction::Schema`, `ConfigSubcommand::Schema`, `ConfigResult::Schema` / `SchemaWritten` variants. Fields without defaults are emitted without a `type` key (any JSON value accepted) since confique's `Meta` does not carry Rust type information directly.
  - **Structured error data API** — New `UnknownKeyInfo` struct (flat `{ key, path, line, source }`) replaces the nested `ClapfigError::UnknownKey`/`UnknownKeys(Vec<ClapfigError>)` shape. Accessor methods `ClapfigError::unknown_keys() -> Option<&[UnknownKeyInfo]>`, `parse_error() -> Option<(&Path, &toml::de::Error, Option<&str>)>`, and `is_strict_violation() -> bool` let callers read error data without pattern-matching on enum variants. Full source text is retained (as `Arc<str>`) on unknown-key and parse errors so renderers can draw snippets. New public export: `UnknownKeyInfo`.
  - **`clapfig::render` module** — Presentation layer separate from `ClapfigError`. `render_plain(&err) -> String` produces ANSI-free output with source snippets and carets, always available. `render_rich(&err) -> String` produces colored, aligned output via [`miette`](https://docs.rs/miette)'s graphical report handler, behind the new `rich-errors` Cargo feature. Both functions return strings — the caller decides where output lands (stderr, log file, TUI pane).
  - **`rich-errors` Cargo feature** — Opt-in `miette` integration for terminal-quality error output. Disabled by default to keep the dependency footprint small.
  - **Configurable layer precedence** — New `Layer` enum and `.layer_order()` builder method allow customizing the merge order of configuration sources. The default order (`Files < Env < Url < Cli`) is preserved when unset. Layers listed later override earlier ones; omitting a layer excludes it from merging entirely. New public export: `Layer`.
  - **URL query parameter layer** — New `.url_query()` builder method parses URL query strings (`port=9090&database.url=pg%3A%2F%2Fprod`) into config overrides. Keys use `.` for nesting, values are percent-decoded and parsed with the same heuristic as env vars. Sits between env vars and CLI overrides in precedence: defaults < files < env < **URL** < CLI. Requires the `url` Cargo feature (`dep:percent-encoding`).
- **Documentation**
  - **Restructured documentation** — Moved the comprehensive user guide from README into crate-level doc comments (published to docs.rs), covering design rationale, trade-offs, and "when to use what" guidance. Added module-level docs to `builder` and `error` modules. Slimmed README from 537 to 126 lines as a landing page with feature list, quick start, and link to docs.rs. ([#13](https://github.com/arthur-debert/clapfig/pull/13))
- **Fixed**
  - **`config set` now validates values before persisting** - Previously, `config set mode garbage` would silently write invalid values to the TOML file, only surfacing errors on the next `load()`. Now `set_in_document` validates both that the key exists in the config schema and that the value is type-compatible (e.g. valid enum variant) by round-trip deserializing into `C::Layer` before writing. Invalid values produce a clear `InvalidValue` error; unknown keys produce `KeyNotFound`. This closes the structural gap where `load()` and `cli_overrides_from()` validated but `config set` did not. ([#9](https://github.com/arthur-debert/clapfig/issues/9))
- **Added**
  - **`ConfigCommand` builder** — Runtime-configurable alternative to `ConfigArgs` for apps that need to rename config subcommands or flags to avoid conflicts with their own CLI. Builder methods: `list_name()`, `gen_name()`, `get_name()`, `set_name()`, `unset_name()`, `scope_long()`, `output_long()`, `output_short()`. Produces the same `ConfigAction` as the derive path, so all downstream logic is shared. ([#11](https://github.com/arthur-debert/clapfig/issues/11))
  - **Named persist scopes** - New `.persist_scope(name, path)` builder method replaces `.persist_path()`. Scopes are named config file targets (e.g. "local", "global") for read/write operations. The first scope added is the default for writes. Scope paths are automatically added to search paths for discovery.
  - **`--scope` CLI flag** - New `--scope <name>` global flag on `ConfigArgs` targets a specific persist scope. Works with all config subcommands: `set`/`unset` write to the named scope; `list`/`get` read from that scope's file only (instead of the merged resolved view).
  - **Scope file operations** - `list_scope_file()` and `get_scope_value()` read entries from individual scope files without going through the full resolve pipeline.
  - **`UnknownScope` error** - New `ClapfigError::UnknownScope` with the invalid scope name and list of available scopes.
  - **`config list`** - New `ConfigAction::List` and `ConfigSubcommand::List` to show all resolved key-value pairs. Bare `app config` (no subcommand) defaults to list. Uses the flatten module to display dotted keys, with `<not set>` for unset optional fields.
  - **Demo application** - `examples/clapfig_demo/` is a runnable sample CLI that exercises all core features: multi-path file search, env var overrides (`CLAPFIG_DEMO__*`), CLI flag mapping (both `cli_overrides_from` and manual `cli_override`), nested config structs, `config gen|get|set|list` subcommands, and ANSI-colored terminal output. Run with `cargo run --example clapfig_demo -- echo`.
  - **Search modes** - New `SearchMode` enum on the builder via `.search_mode()`. `Merge` (default) deep-merges all found config files; `FirstMatch` uses only the highest-priority file found ("find my config" pattern).
  - **Ancestor walk** - New `SearchPath::Ancestors(Boundary)` variant walks up from the current working directory to discover config files. `Boundary::Root` walks to the filesystem root; `Boundary::Marker(".git")` stops at a project boundary (inclusive). Expands inline in the search path list, composable with other variants.
  - **Feature-gated clap dependency** - `clap` is now an optional dependency behind the `clap` Cargo feature (enabled by default). The `cli` module, `ConfigArgs`, and `ConfigSubcommand` are only compiled when the feature is active. Use `default-features = false` to use clapfig without pulling in clap.
- **Changed**
  - **`ClapfigError` error variants restructured** — The single-key `ClapfigError::UnknownKey { key, path, line }` variant was removed; all strict-mode violations now flow through `ClapfigError::UnknownKeys(Vec<UnknownKeyInfo>)` (a one-element vector for the single-key case). `ClapfigError::ParseError` now wraps `Box<toml::de::Error>` (to keep the enum variant small) and carries a new `source_text: Option<Arc<str>>` field with the retained file contents for snippet rendering. Callers pattern-matching on these variants will need to update; the new accessor methods (`unknown_keys()`, `parse_error()`) are a more stable alternative.
  - `.persist_path()` replaced by `.persist_scope(name, path)`. Scopes are named targets; the first added is the default.
  - `ConfigAction` variants `List`, `Get`, `Set`, `Unset` now carry `scope: Option<String>`.
  - `config set`/`unset` require at least one `.persist_scope()`. Omitting returns `ClapfigError::NoPersistPath`.
  - `load_config_files` now takes a `SearchMode` parameter.
  - New public exports: `Boundary`, `SearchMode`.
  - Crate and README documentation restructured to clarify that the core API is framework-agnostic and clap is an optional adapter.

## [0.1.0] - 2026-02-11

- **Added**
  - **Struct-driven config** - Define settings as a Rust struct with `#[config(default)]` and doc comments via confique
  - **Layered merge** - defaults < config files < env vars < CLI flags, every layer sparse
  - **Multi-path file search** - Platform config dir, home subdir, cwd, or explicit paths
  - **Prefix-based env vars** - `MYAPP__DATABASE__URL` maps to `database.url` with heuristic type parsing
  - **Clap CLI overrides** - `.cli_override("key", value)` maps any clap arg to any config key
  - **Strict mode** - Unknown keys in config files error with file path, key name, and line number (on by default)
  - **Template generation** - `config gen` emits a commented TOML template from struct doc comments
  - **Config subcommand** - Drop-in `config gen|get|set` commands for clap
  - **Persistence** - `config set` patches values in place via `toml_edit`, preserving comments and formatting
